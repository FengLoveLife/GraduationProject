### 1.2 为什么销售数据中心是核心？

**数据资产价值链**：

```
销售数据 → 历史销量分析 → 销量预测 → 补货决策 → 库存优化 → 成本降低 → 利润提升
```

**没有销售数据会发生什么？**

| 模块 | 没有销售数据的后果 |
|------|------------------|
| 预测模块 | 无数据可训练，预测模型无法建立 |
| 补货模块 | 只能凭经验拍脑袋进货，容易造成缺货或积压 |
| 库存模块 | 无法知道合理的安全库存应该设多少 |

### 1.3 销售数据中心的核心职责

1. **数据采集**：记录每一笔销售交易的完整信息
2. **数据存储**：以合理的结构持久化保存销售数据
3. **数据关联**：将销售数据与商品、时间、因子等维度关联
4. **数据供给**：为预测和补货模块提供高质量的数据基础

## 二、数据库表设计详解

### 2.1 为什么采用主从表结构？

**方案对比**：

| 方案 | 结构 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 单表 | 一个表存所有 | 简单 | 数据冗余、难以扩展 | 每天只记录汇总数据 |
| **主从表** | 订单主表 + 明细从表 | 规范、灵活、易扩展 | 稍复杂 | **推荐** |

**选择主从表的原因**：

```
真实销售场景：
顾客买了 3 件商品（1瓶可乐、2包薯片、1袋面包）

如果用单表：
┌────────────────────────────────────────────────────────────┐
│ 一条记录：订单号=001, 商品=可乐,薯片,面包, 数量=3, 金额=45  │
│ 问题：商品信息混在一起，无法按商品统计销量                     │
└────────────────────────────────────────────────────────────┘

如果用主从表：
主表（订单级别）：
┌─────────────────────────────────────────┐
│ 订单号=001, 总数量=4, 总金额=45, 时间=... │
└─────────────────────────────────────────┘

从表（商品级别）：
┌──────────────────────────────────────────────────────────┐
│ 订单号=001, 商品=可乐,   单价=3,  数量=1, 小计=3          │
│ 订单号=001, 商品=薯片,   单价=8,  数量=2, 小计=16         │
│ 订单号=001, 商品=面包,   单价=6,  数量=1, 小计=6          │
└──────────────────────────────────────────────────────────┘

优势：可以按商品统计销量，可以按订单查询明细，结构清晰
```

### 2.2 销售订单主表设计 (`sales_order`)

```sql
CREATE TABLE `sales_order` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `order_no` VARCHAR(32) NOT NULL COMMENT '订单编号',
    `total_amount` DECIMAL(12,2) NOT NULL DEFAULT 0.00 COMMENT '订单总金额',
    `total_quantity` INT NOT NULL DEFAULT 0 COMMENT '商品总数量',
    `payment_type` TINYINT NOT NULL DEFAULT 1 COMMENT '支付方式：1现金 2微信 3支付宝',
    `sale_date` DATE NOT NULL COMMENT '销售日期',
    `sale_time` DATETIME NOT NULL COMMENT '销售时间',
    `operator` VARCHAR(50) DEFAULT NULL COMMENT '操作员',
    `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_order_no` (`order_no`),
    KEY `idx_sale_date` (`sale_date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='销售订单主表';
```

**字段设计思路**：

| 字段 | 为什么需要 | 设计考量 |
|------|-----------|---------|
| `order_no` | 业务唯一标识 | 便于查询、展示、对账 |
| `total_amount` | 订单级别统计 | 冗余存储，避免每次计算 |
| `total_quantity` | 订单级别统计 | 同上 |
| `payment_type` | 支付方式分析 | 可用于分析支付偏好 |
| **`sale_date`** | **关键！关联因子表** | 只存日期，用于统计和关联 |
| `sale_time` | 精确时间 | 用于记录具体时间点 |

### 2.3 销售订单明细表设计 (`sales_order_item`)

```sql
CREATE TABLE `sales_order_item` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `order_id` BIGINT NOT NULL COMMENT '关联订单主表ID',
    `order_no` VARCHAR(32) NOT NULL COMMENT '订单编号（冗余）',
    `product_id` BIGINT NOT NULL COMMENT '商品ID',
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码（冗余）',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称（冗余）',
    `category_id` BIGINT DEFAULT NULL COMMENT '分类ID（冗余）',
    `category_name` VARCHAR(50) DEFAULT NULL COMMENT '分类名称（冗余）',
    `unit_price` DECIMAL(10,2) NOT NULL COMMENT '销售单价',
    `quantity` INT NOT NULL COMMENT '销售数量',
    `subtotal_amount` DECIMAL(12,2) NOT NULL COMMENT '小计金额',
    `sale_date` DATE NOT NULL COMMENT '销售日期（冗余）',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_order_id` (`order_id`),
    KEY `idx_product_id` (`product_id`),
    KEY `idx_sale_date` (`sale_date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='销售订单明细表';
```

**为什么要冗余存储商品名称、分类名称等？**

```
场景：查询某天的销售明细并展示

方案A（不冗余）：
1. 查 sales_order_item 表
2. 对每条记录，再查 product 表获取商品名称
3. 对每条记录，再查 category 表获取分类名称
→ 多次 JOIN，性能差

方案B（冗余存储）：
1. 查 sales_order_item 表
2. 直接展示 product_name、category_name
→ 单表查询，性能好

注意：冗余的是"下单时刻的信息"，即使后来商品改名了，
      销售记录应该保持下单时的状态，这是正确的业务逻辑！
```

### 2.4 日历影响因子表设计 (`calendar_factor`)

**设计原则**：面向中小型超市，保留核心因子，避免过度设计。

```sql
CREATE TABLE `calendar_factor` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `date` DATE NOT NULL COMMENT '日期（唯一）',

    -- 时间特征
    `day_of_week` TINYINT NOT NULL COMMENT '星期几 1-7（1=周一，7=周日）',
    `is_weekend` TINYINT NOT NULL DEFAULT 0 COMMENT '是否周末：0否 1是',

    -- 节假日特征
    `is_holiday` TINYINT NOT NULL DEFAULT 0 COMMENT '是否节假日：0否 1是',
    `holiday_name` VARCHAR(50) DEFAULT NULL COMMENT '节假日名称：春节、国庆等',

    -- 天气特征（简化）
    `weather` VARCHAR(20) DEFAULT NULL COMMENT '天气：晴/多云/雨/雪',

    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_date` (`date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='日历影响因子表';
```

**为什么这样设计？**

| 字段 | 保留原因 |
|------|---------|
| `day_of_week` | 周末销量通常高于工作日，是核心因子 |
| `is_weekend` | 便于快速筛选周末数据 |
| `is_holiday` | 节假日对销量影响显著（如春节、国庆） |
| `holiday_name` | 区分不同节假日的影响程度 |
| `weather` | 天气对出行和购买意愿有影响 |

**为什么简化？**

| 去掉的字段 | 理由 |
|-----------|------|
| 温度（高/低/平均） | 中小超市不需要这么精细的温度数据 |
| 湿度、风力 | 影响较小，可忽略 |
| 特殊事件 | 中小超市场景少，维护成本高 |
| 促销活动标识 | 可单独管理，或在销售录入时标记 |

**天气简化为4类**：

| 天气 | 对销量的影响 |
|------|-------------|
| 晴 | 正常或偏高 |
| 多云 | 正常 |
| 雨 | 可能下降 |
| 雪 | 可能下降较多 |

**数据来源与维护**：

| 因子 | 来源 | 维护方式 |
|------|------|---------|
| 时间特征 | 系统计算 | 自动生成 |
| 节假日 | 国务院发布 | 每年导入一次 |
| 天气 | 天气API | 定时任务每日同步 |

## 三、影响因子体系设计

### 3.1 核心影响因子

```
┌─────────────────────────────────────────────────────────────────────┐
│                      销量影响因子（简化版）                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    1. 时间因子（核心）                        │   │
│  ├─────────────────────────────────────────────────────────────┤   │
│  │  • 星期几        周末销量通常高于工作日                       │   │
│  │  • 是否周末      快速判断周末标识                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    2. 节假日因子（核心）                      │   │
│  ├─────────────────────────────────────────────────────────────┤   │
│  │  • 是否节假日    春节、国庆等销量波动大                       │   │
│  │  • 节假日名称    不同节假日影响程度不同                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    3. 天气因子（核心）                        │   │
│  ├─────────────────────────────────────────────────────────────┤   │
│  │  • 天气类型      晴/多云/雨/雪，影响出行和购买意愿            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    4. 商品历史因子（预测时计算）              │   │
│  ├─────────────────────────────────────────────────────────────┤   │
│  │  • 历史销量      过去7天/30天平均销量                         │   │
│  │  • 销量趋势      环比变化                                     │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 各因子对销量的影响分析

| 因子 | 影响方向 | 影响程度 |
|------|---------|---------|
| **周末** | 销量↑ | +20%~50% |
| **春节/国庆** | 销量↑↑ | +50%~200% |
| **晴天** | 销量正常或↑ | 基准 |
| **雨天** | 销量↓ | -10%~20% |
| **雪天** | 销量↓↓ | -20%~40% |

### 3.3 为什么不考虑更多因子？

| 因子 | 不考虑的原因 |
|------|-------------|
| 温度 | 中小超市不需要精确到温度，天气类型足够 |
| 时段 | 我们做的是**日销量预测**，不是时段预测 |
| 促销活动 | 维护成本高，初期可简化处理 |
| 特殊事件 | 中小超市场景少，ROI低 |

## 四、预测模块如何运作

### 4.1 预测的基本原理

```
┌─────────────────────────────────────────────────────────────────────┐
│                         销量预测原理                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   输入                          模型                    输出        │
│  ┌───────┐                    ┌───────┐              ┌───────┐    │
│  │历史销量│ ──────────────────▶│ 预测  │─────────────▶│未来销量│    │
│  │影响因子│                    │ 模型  │              │预测值  │    │
│  └───────┘                    └───────┘              └───────┘    │
│                                                                     │
│  举例：                                                             │
│  输入：过去30天可乐的每日销量 + 每天的天气/是否周末/是否节假日        │
│  模型：发现规律 - 周末销量+30%，雨天销量-15%，节假日销量+50%         │
│  输出：预测明天（周六+晴天+无节假日）销量 = 基准销量 × 1.3 × 1.0 × 1.0│
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 训练数据如何准备

```sql
-- 查询训练数据：每个商品每天的销量 + 影响因子
SELECT
    s.sale_date,
    s.product_id,
    s.product_name,
    SUM(s.quantity) AS daily_sales,      -- 每日销量（目标值）

    -- 时间因子
    c.day_of_week,
    c.is_weekend,

    -- 节假日因子
    c.is_holiday,
    c.holiday_name,

    -- 天气因子
    c.weather

FROM sales_order_item s
LEFT JOIN calendar_factor c ON s.sale_date = c.date
WHERE s.product_id = ?
  AND s.sale_date BETWEEN '2024-01-01' AND '2024-12-31'
GROUP BY s.sale_date, s.product_id, s.product_name
ORDER BY s.sale_date;
```

**训练数据示例**：

| sale_date | product_name | daily_sales | day_of_week | is_weekend | is_holiday | weather |

| 2024-01-01 | 可乐 | 45 | 1 | 0 | 1 | 晴 |
| 2024-01-02 | 可乐 | 32 | 2 | 0 | 0 | 多云 |
| 2024-01-06 | 可乐 | 58 | 6 | 1 | 0 | 雨 |
| 2024-01-07 | 可乐 | 62 | 7 | 1 | 0 | 晴 |
| ... | ... | ... | ... | ... | ... | ... |

### 4.3 预测流程

```
步骤1：获取历史数据
┌─────────────────────────────────────────────────────────────┐
│  从 sales_order_item 获取过去N天的每日销量                    │
│  从 calendar_factor 获取对应日期的影响因子                    │
└─────────────────────────────────────────────────────────────┘
                          ↓
步骤2：特征处理
┌─────────────────────────────────────────────────────────────┐
│  计算移动平均、环比变化等衍生特征                              │
│  对天气等分类变量进行编码                                      │
└─────────────────────────────────────────────────────────────┘
                          ↓
步骤3：模型预测
┌─────────────────────────────────────────────────────────────┐
│  输入：未来N天的因子（天气预报 + 节假日信息）                  │
│  输出：每个商品未来N天的预测销量                               │
└─────────────────────────────────────────────────────────────┘
                          ↓
步骤4：结果存储
┌─────────────────────────────────────────────────────────────┐
│  将预测结果存入 forecast_result 表                            │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 预测结果表设计

```sql
CREATE TABLE `forecast_result` (
    `id` BIGINT NOT NULL AUTO_INCREMENT,
    `forecast_date` DATE NOT NULL COMMENT '预测执行日期',
    `product_id` BIGINT NOT NULL COMMENT '商品ID',
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称',
    `target_date` DATE NOT NULL COMMENT '预测目标日期',
    `predicted_quantity` DECIMAL(10,2) NOT NULL COMMENT '预测销量',
    `actual_quantity` DECIMAL(10,2) DEFAULT NULL COMMENT '实际销量（发生后填充）',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_target_date` (`target_date`),
    KEY `idx_product_id` (`product_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='销量预测结果表';
```

## 五、智能进货模块如何运作

### 5.1 进货决策的核心逻辑

```
┌─────────────────────────────────────────────────────────────────────┐
│                         补货决策公式                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  建议补货量 = 预测销量 - 当前库存 + 安全库存                         │
│                                                                     │
│  举例：                                                             │
│  • 当前库存：50 瓶                                                  │
│  • 未来7天预测销量：20+25+18+22+30+35+28 = 178 瓶                   │
│  • 安全库存：30 瓶                                                  │
│  • 建议补货量 = 178 - 50 + 30 = 158 瓶                              │
│                                                                     │
│  如果结果 ≤ 0，说明库存充足，不需要补货                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 补货建议表设计

```sql
CREATE TABLE `restocking_suggestion` (
    `id` BIGINT NOT NULL AUTO_INCREMENT,
    `suggestion_date` DATE NOT NULL COMMENT '建议生成日期',
    `product_id` BIGINT NOT NULL COMMENT '商品ID',
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称',
    `current_stock` INT NOT NULL COMMENT '当前库存',
    `safety_stock` INT NOT NULL COMMENT '安全库存',
    `predicted_sales_7d` INT DEFAULT NULL COMMENT '未来7天预测销量',
    `suggested_quantity` INT NOT NULL COMMENT '建议补货数量',
    `urgency_level` TINYINT NOT NULL DEFAULT 2 COMMENT '紧急程度：1紧急 2一般 3低',
    `status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0待处理 1已采纳 2已忽略',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_suggestion_date` (`suggestion_date`),
```

## 总结

**销售数据中心是整个系统的数据基石**：

1. 它记录了超市的核心业务数据——销售流水
2. 它为预测模块提供了训练数据（历史销量）
3. 它关联了日历因子表，使预测可以考虑核心影响因子
4. 预测模块的输出又支撑了智能补货模块的决策

**数据库表设计要点**：

1. `sales_order` + `sales_order_item` 主从表结构，灵活且规范
2. `calendar_factor` 只保留核心因子（时间/节假日/天气），避免过度设计
3. 冗余存储商品名称等字段，提升查询性能
