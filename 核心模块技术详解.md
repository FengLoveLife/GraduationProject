## 一、销量预测模块

### 1.1 整体架构

销量预测采用 **Python 微服务 + Java 后端分离** 的设计：

```
Java 后端
    │  HTTP POST /forecast/run
    ▼
Python FastAPI 服务（端口 8000）
    │  1. 读取历史销售数据 + 日历因子
    │  2. 特征工程（构造 16 维特征向量）
    │  3. LightGBM 滚动预测（逐日推进）
    │  4. 预测结果写入 forecast_result 表
    ▼
MySQL forecast_result 表
    ▼
Java ForecastResultServiceImpl
    │  读取预测结果 + 商品库存 + 历史日均
    ▼
前端展示（预测看板 / 销量预测明细）
```

**为什么用独立 Python 服务？**

机器学习生态在 Python 中最为成熟。LightGBM、Pandas、NumPy 等库在 Python 中一行代码即可使用，Java 中需要繁复的 JNI 绑定或额外封装。独立服务带来另一个好处：**模型可以单独升级**，无需改动 Java 业务代码。

### 1.2 LightGBM 算法选型

#### 为什么选 LightGBM

| 方案 | 问题 |
|------|------|
| ARIMA | 只能单变量，无法引入节假日、天气等外部因素 |
| Prophet | 适合年度趋势预测，不适合商品级日销量 |
| LSTM | 数据量要求高，调试成本大，过拟合风险高 |
| XGBoost | 功能类似，但训练速度慢 3-10 倍，类别特征需手动 One-Hot 编码 |
| **LightGBM** | 训练速度快、内存占用低、原生支持类别特征、API 简洁 |

#### LightGBM 核心原理

LightGBM 是基于 **梯度提升决策树（GBDT）** 的框架，采用两项关键创新：

**1. 直方图算法（Histogram Algorithm）**  
将连续特征值离散化成直方图区间，减少每次分裂时的计算量，使训练速度比传统 GBDT 快 3-10 倍。

**2. 按叶子生长（Leaf-wise）**  
每次只分裂误差增益最大的叶子节点，相比传统"按层生长"能用更少的树达到更低的误差。

```
按层生长（传统）：       按叶子生长（LightGBM）：
      根                       根
     /  \                     /  \
    A    B                   A    B
   / \  / \                 / \
  C  D E  F                C   D   ← 只分裂误差最大的 A
（全部分裂，计算量大）    （精准分裂，效率更高）
```

**最终预测 = 所有决策树预测值的加权求和**，每棵新树专门纠正前面所有树的残差。

**目标变量**：当天实际销量 `quantity`

#### 缺失日期填充（关键修复）

当某天某商品销量为 0 时，SQL `GROUP BY` 不会返回该记录。若不处理，滑动窗口取到的是"过去 7 条有销量的记录"而非"过去 7 天"，特征错误。

解决方案：生成**商品 × 日期**笛卡尔积，将缺失日期的销量填充为 0，保证时间序列连续性。

```python
# 生成完整日期序列
date_range = pd.date_range(start=start_date, end=end_date, freq='D')

# 商品 × 日期 笛卡尔积
product_date_combinations = pd.DataFrame(
    list(cartesian_product(products['product_code'], date_range)),
    columns=['product_code', 'date']
)

# 合并销量，缺失填 0
filled_df = product_date_combinations.merge(
    df[['date', 'product_code', 'quantity']],
    on=['date', 'product_code'], how='left'
)
filled_df['quantity'] = filled_df['quantity'].fillna(0).astype(int)
```

### 1.4 滚动预测（Rolling Prediction）

#### 为什么需要滚动预测

LightGBM 模型使用"前 7 天销量"作为特征。预测第 1 天可以用真实历史数据，但预测第 2 天时需要第 1 天的销量，而此时第 1 天还没有真实数据——需要用预测值代替，然后依次推进。

#### 滚动预测流程

```
已知：真实历史 [D-7, D-6, D-5, D-4, D-3, D-2, D-1]

预测 D+1：
  features = [D-6, D-5, D-4, D-3, D-2, D-1, D] + 日历因子
  pred_D+1 = model.predict(features)

预测 D+2：
  features = [D-5, D-4, D-3, D-2, D-1, D, pred_D+1] + 日历因子
  pred_D+2 = model.predict(features)

...以此类推，连续推进 7 天
```

预测结果通过 `ON DUPLICATE KEY UPDATE` 写入 `forecast_result` 表，同一商品同一日期的预测可安全覆盖。

### 1.6 预测结果展示（Java 侧）

Java `ForecastResultServiceImpl` 在读取预测数据时：

2. 从 `sales_order_item` 表批量计算过去 30 天的历史日均销量

4. 用动态黄灯线公式计算三灯状态

**三灯判断逻辑（Java）：**

```java
private String calculateStockStatus(Integer currentStock, Integer safetyStock,
                                     Double dailyAvg, Integer restockCycleDays) {
    if (currentStock <= safetyStock) return "warning";          // 红灯
    double yellowLine = safetyStock + dailyAvg * restockCycleDays * 0.3;
    if (currentStock <= yellowLine) return "needPurchase";      // 黄灯
    return "sufficient";                                        // 绿灯
}
```

### 2.2 三灯策略

#### 红灯（必须进货）

**触发条件**：`当前库存 ≤ 安全库存`

库存已跌破安全水位，随时面临缺货风险，必须立即补货。

#### 黄灯（顺带进货）

**触发条件**：`安全库存 < 当前库存 ≤ 黄灯线`

```
黄灯线 = 安全库存 + 日均销量 × 补货周期天数 × 30%
```

含义：当前库存只够撑过周期的 30%，既然同分类已有红灯商品触发进货，顺带一起补，避免短期内再次进货。

#### 绿灯（不进货）

**触发条件**：`当前库存 > 黄灯线`

库存健康，无需进货。

#### 策略优势

相比简单的"库存低于 X 天就进货"，红黄绿三灯具有以下优势：

1. **防止促销误判**：某商品因促销爆单导致库存骤降，但其他商品库存充足，系统只补缺货商品，不会把全分类都补一遍
2. **智能凑单**：黄灯商品顺带进货，减少采购频次和物流成本
3. **动态黄灯线**：不同补货周期的商品黄灯触发时机不同，生鲜（3天周期）和粮油（20天周期）的阈值自动适配

**场景举例：**

| 商品 | 安全库存 | 日均销量 | 补货周期 | 黄灯线 | 当前库存 | 状态 |
|------|--------|---------|--------|--------|--------|------|
| 矿泉水 | 10 | 50 | 7天 | 10+50×7×30%=115 | 8 | 🔴红灯 |
| 橙汁 | 10 | 30 | 7天 | 10+30×7×30%=73 | 50 | 🟡黄灯 |
| 绿茶 | 10 | 20 | 7天 | 10+20×7×30%=52 | 80 | 🟢绿灯 |

→ 此次进货：矿泉水（必须）+ 橙汁（顺带），绿茶不动。

### 2.3 三档日均权重（与预测模块联动）

日均销量不能单纯用 AI 预测值，因为补货周期越长，短期 AI 预测的噪声放大效应越明显。系统根据补货周期分三档混合：

| 补货周期 | AI 预测权重 | 历史均值权重 | 适用场景 |
|--------|-----------|-----------|--------|
| ≤ 7 天  | 100% | 0% | 生鲜、果蔬等快消品，AI 预测最可靠 |
| 7-14 天 | 60% | 40% | 饮料、日化等中速周转品 |
| ≥ 15 天 | 30% | 70% | 粮油、日百等慢周转品，历史均值更稳健 |

**数据来源：**
- AI 预测日均：`forecast_result` 表未来 7 天预测值均值
- 历史日均：`sales_order_item` 表过去 30 天实际销量均值

**Java 实现（`blendDailySales`）：**

```java
private BigDecimal blendDailySales(Long productId, Integer restockCycleDays,
                                   Map<Long, BigDecimal> historicalMap,
                                   Map<Long, BigDecimal> predictedMap) {
    BigDecimal hist = historicalMap.getOrDefault(productId, BigDecimal.ZERO);
    BigDecimal pred = predictedMap.getOrDefault(productId, BigDecimal.ZERO);

    if (restockCycleDays <= 7) {
        return pred.compareTo(BigDecimal.ZERO) > 0 ? pred : hist;
    }
    if (restockCycleDays < 15) {
        if (pred.compareTo(BigDecimal.ZERO) == 0) return hist;
        return pred.multiply(BigDecimal.valueOf(0.6))
                .add(hist.multiply(BigDecimal.valueOf(0.4)))
                .setScale(2, RoundingMode.HALF_UP);
    }
    // ≥ 15 天
    if (pred.compareTo(BigDecimal.ZERO) == 0) return hist;
    return pred.multiply(BigDecimal.valueOf(0.3))
            .add(hist.multiply(BigDecimal.valueOf(0.7)))
            .setScale(2, RoundingMode.HALF_UP);
}
```

前端销量预测页面使用完全相同的权重公式，保证两个模块的日均销量计算结果一致。

### 2.4 建议进货量计算

```
目标库存 = 日均销量 × 补货周期天数 + 安全库存
建议进货量 = MAX(0, 目标库存 - 当前库存)
```

**含义**：进货后库存恰好达到"能够支撑整个周期销售 + 安全库存缓冲"的水位。

**数值举例**（矿泉水，补货周期 7 天）：
- 日均销量（预测）= 50 瓶
- 安全库存 = 10 瓶
- 目标库存 = 50 × 7 + 10 = 360 瓶
- 当前库存 = 8 瓶
- **建议进货量 = 360 - 8 = 352 瓶**

### 2.5 安全库存科学计算

安全库存不是拍脑袋设定，而是基于统计学公式，以 **95% 服务水平**（Z = 1.65）计算：

```
安全库存 = CEIL(1.65 × 日销量标准差 × √补货周期天数)
```

**公式含义**：

- `日销量标准差` 反映需求波动程度，波动越大安全库存越高
- `√补货周期` 体现周期越长、累积风险越大（统计学中的"时间放大效应"）
- `1.65` 是正态分布 95% 分位数，确保 100 次补货周期中最多 5 次缺货

**约束条件**：
- 下限 = `0.8 × 日均销量`：防止安全库存过低（如果标准差极小）
- 上限 = `3 × 日均销量`：防止畸高（如果标准差异常大）

安全库存基于 2025-09-01 至 2026-04-11 约 7 个月的真实销售数据计算，通过 `db/safety_stock.sql` 批量写入数据库，共覆盖 101 个在售商品。

### 2.6 N+1 查询优化

进货建议生成时需要为每个商品计算日均销量。若逐个商品查数据库，101 个商品将产生约 400 条 SQL。

**优化方案：批量预加载**

```java
// 仅 2 条 SQL，替代 400 条
Map<Long, BigDecimal> historicalDailyMap = buildHistoricalDailyMap(30);
Map<Long, BigDecimal> predictedDailyMap  = buildPredictedDailyMap(7);
```

对应的批量 SQL（以历史日均为例）：

```sql
SELECT product_id,
       SUM(quantity)               as totalQuantity,
       COUNT(DISTINCT sale_date)   as salesDays
FROM sales_order_item
WHERE sale_date >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
GROUP BY product_id
```

结果存入 `Map<Long, BigDecimal>`（productId → 日均），后续循环从 Map 取值，**零数据库查询**。

**整个进货建议生成流程固定只有 4 条 SQL**：
1. 批量历史日均（GROUP BY）
2. 批量预测日均（GROUP BY）
3. 查询所有红灯商品
4. 查询所有非红灯商品

## 三、两模块的数据联动

销量预测和智能进货共享同一套日均计算权重，确保**预测看板**与**进货建议**中展示的日均销量和库存状态完全一致：

```
forecast_result 表（Python 写 → Java 读）
        ↓
ForecastResultServiceImpl.calculateStockStatus()
    → 三灯判断（红/黄/绿）
    → 用于：预测看板、销量预测明细

PurchaseSuggestionServiceImpl.blendDailySales()
    → 同样的三档权重公式
    → 用于：进货建议量计算、黄灯线判断
```

**一致性保证**：两处都取 `forecast_result` 未来 7 天预测均值作为 AI 预测日均，都取 `sales_order_item` 过去 30 天作为历史日均，权重分档完全相同。用户在预测看板看到"库存告急"的商品，在进货建议中必然出现为红灯商品。
