## 一、业务背景回顾

### 1.1 核心算法

根据之前确定的方案，智能进货采用**双水位模型 + 红黄绿灯机制**：

```
目标库存 = 日均销量 × 补货周期 + 安全库存
建议进货量 = 目标库存 - 当前库存
```

**红黄绿灯规则**：
- 🔴 **红灯**：当前库存 ≤ 安全库存 → 必须补货
- 🟡 **黄灯**：安全库存 < 当前库存 ≤ 安全库存 + 周期销量×30% → 顺带补货
- 🟢 **绿灯**：库存健康，不补货

### 1.2 分类补货周期

| 分类 | 补货周期(天) | 原因 |
|------|-------------|------|
| 生鲜果蔬 | 3 | 保质期短，快进快出 |
| 饮料 | 14 | 销量稳定 |
| 休闲零食 | 14 | 销量稳定 |
| 粮油调味 | 30 | 销量慢 |
| 日用百货 | 30 | 保质期长 |

## 二、需要哪些表？

### 2.1 表结构评估

| 表名 | 是否必要 | 说明 |
|------|----------|------|
| `purchase_suggestion` 进货建议表 | ✅ 必要 | 存储系统生成的进货建议 |
| `purchase_order` 进货单主表 | ✅ 必要 | 用户确认后的正式进货单 |
| `purchase_order_item` 进货单明细表 | ✅ 必要 | 进货单的商品明细 |
| `supplier` 供应商表 | ❌ 不需要 | 毕设阶段可简化，供应商信息直接记录在进货单备注 |
| `category_restock_batch` 分类进货批次表 | ❌ 不需要 | 过度设计，batch_no 字段即可满足 |

### 2.2 最终方案

**需要 3 张核心表 + 修改分类表**

```
┌─────────────────────┐
│   product_category  │ ←── 新增 restock_cycle_days 字段
└──────────┬──────────┘
           │ 分类配置决定补货周期
           ▼
┌─────────────────────┐       ┌─────────────────────┐
│       product       │       │   forecast_result   │
│     (商品/库存)     │       │    (预测结果)       │
└──────────┬──────────┘       └──────────┬──────────┘
           │                             │
           │ 库存信息                     │ 预测销量
           └─────────────┬───────────────┘
                         ▼
┌─────────────────────────────────────────────┐
│           purchase_suggestion               │
│           (进货建议表)                       │
└────────────────────┬────────────────────────┘
                     │ 用户确认
                     ▼
┌─────────────────────────────────────────────┐
│           purchase_order (主表)              │
│           purchase_order_item (明细表)       │
└─────────────────────────────────────────────┘
```

## 三、表结构详细设计

### 3.1 修改商品分类表

在分类表上新增补货周期字段：

```sql
-- 新增字段
ALTER TABLE `product_category`
ADD COLUMN `restock_cycle_days` INT DEFAULT 7 COMMENT '补货周期(天)' AFTER `sort_order`;
```

**说明**：
- 不同分类设置不同周期（生鲜3天、饮料14天等）
- 系统根据此字段计算目标库存

---

### 3.2 进货建议表 `purchase_suggestion`

```sql
-- 进货建议表
CREATE TABLE `purchase_suggestion` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 关联信息
    `product_id` BIGINT NOT NULL COMMENT '商品ID',
    `category_id` BIGINT NOT NULL COMMENT '分类ID',

    -- ========== 冗余字段（减少JOIN查询） ==========
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称',
    `category_name` VARCHAR(50) NOT NULL COMMENT '分类名称',
    `purchase_price` DECIMAL(10,2) DEFAULT NULL COMMENT '进货单价',

    -- ========== 预测基础数据 ==========
    `predicted_quantity` INT NOT NULL COMMENT '预测销量',
    `daily_sales` DECIMAL(10,2) DEFAULT 0 COMMENT '日均销量(预测值或近7天均值)',

    -- ========== 库存状态 ==========
    `current_stock` INT NOT NULL COMMENT '当前库存',
    `safety_stock` INT NOT NULL COMMENT '安全库存',
    `target_stock` INT NOT NULL COMMENT '目标库存',

    -- ========== 灯位状态 ==========
    `light_status` TINYINT DEFAULT 1 COMMENT '灯位: 1红灯(必须补货) 2黄灯(顺带补货)',

    -- ========== 进货建议 ==========
    `suggested_quantity` INT NOT NULL COMMENT '系统建议进货量',
    `adjusted_quantity` INT DEFAULT NULL COMMENT '用户调整后的进货量',
    `final_quantity` INT NOT NULL COMMENT '最终进货量',

    -- ========== 状态管理 ==========
    `status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态: 0待处理 1已生成进货单 2已忽略',
    `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注',

    -- ========== 审计字段 ==========
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

    PRIMARY KEY (`id`),
    KEY `idx_category_id` (`category_id`),
    KEY `idx_status` (`status`),
    KEY `idx_light_status` (`light_status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='进货建议表';
```

**字段说明**：

| 字段 | 来源 | 说明 |
|------|------|------|
| `predicted_quantity` | forecast_result | 预测销量 |
| `daily_sales` | 计算值 | 日均销量（用于计算目标库存）|
| `current_stock` | product 表 | 当前库存快照 |
| `safety_stock` | product 表 | 安全库存快照 |
| `target_stock` | 计算值 | 日均销量 × 补货周期 + 安全库存 |
| `light_status` | 计算值 | 1=红灯，2=黄灯 |
| `suggested_quantity` | 计算值 | 目标库存 - 当前库存 |
| `adjusted_quantity` | 用户输入 | 用户可调整进货量 |
| `final_quantity` | 计算值 | 取 adjusted_quantity 或 suggested_quantity |

### 3.3 进货单主表 `purchase_order`

```sql
-- 进货单主表
CREATE TABLE `purchase_order` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `order_no` VARCHAR(32) NOT NULL COMMENT '进货单号',

    -- ========== 订单信息 ==========
    `total_quantity` INT NOT NULL DEFAULT 0 COMMENT '商品总数量',
    `total_amount` DECIMAL(12,2) NOT NULL DEFAULT 0.00 COMMENT '进货总金额',

    -- ========== 时间信息 ==========
    `order_date` DATE NOT NULL COMMENT '下单日期',
    `expected_date` DATE DEFAULT NULL COMMENT '预计到货日期',
    `actual_arrival_date` DATE DEFAULT NULL COMMENT '实际到货日期',

    -- ========== 状态管理 ==========
    `status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态: 0待确认 1已下单 2已完成 3已取消',

    -- ========== 其他 ==========
    `remark` VARCHAR(500) DEFAULT NULL COMMENT '备注',
    `operator` VARCHAR(50) DEFAULT NULL COMMENT '操作人',

    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_order_no` (`order_no`),
    KEY `idx_order_date` (`order_date`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='进货单主表';
```

**状态流转**：

```
待确认(0) → 已下单(1) → 已完成(2)
    ↓
已取消(3)
```

**极简设计说明**：
- 不设计供应商表，供应商信息可写在备注里
- 不设计"部分到货"状态，简化业务流程
- 到货日期字段保留，方便后续扩展

## 四、极简设计原则说明

### 4.1 为什么不设计供应商表？

| 方案 | 评估 |
|------|------|
| 设计供应商表 | 需要供应商管理页面、增删改查、关联商品...工作量大 |
| 写在备注里 | 简单直接，毕设够用 |

**毕设阶段，供应商信息直接写在进货单备注即可**，如："供应商：XX批发部，电话：138xxxx"

### 4.2 为什么不需要进货批次表？

之前的方案设计了 `category_restock_batch` 表来记录每次检查的结果。

**极简方案**：不需要这张表。

- 进货建议表 `purchase_suggestion` 本身就是每次检查的结果
- 按创建时间查询就能知道什么时候生成的建议
- 没必要额外维护一张批次表

### 4.3 为什么保留 light_status 字段？

前端展示时，红灯和黄灯需要不同的视觉样式：

```
🔴 红灯商品 → 红色标签 → "必须补货"
🟡 黄灯商品 → 黄色标签 → "顺带补货"
```

保留这个字段，前端可以根据灯位状态显示不同颜色。

## 五、业务流程 SQL 示例

### 5.1 生成进货建议

```sql
-- Step 1: 找出红灯商品（必须补货）
INSERT INTO purchase_suggestion (
    product_id, category_id,
    product_code, product_name, category_name, purchase_price,
    predicted_quantity, daily_sales,
    current_stock, safety_stock, target_stock,
    light_status, suggested_quantity, final_quantity, status
)
SELECT
    p.id,
    p.category_id,
    p.product_code,
    p.name,
    pc.name,
    p.purchase_price,
    fr.predicted_quantity,
    fr.predicted_quantity AS daily_sales,  -- 简化：用预测值作为日均销量
    p.stock,
    p.safety_stock,
    fr.predicted_quantity * COALESCE(pc.restock_cycle_days, 7) + p.safety_stock,
    1,  -- 红灯
    GREATEST(0, fr.predicted_quantity * COALESCE(pc.restock_cycle_days, 7) + p.safety_stock - p.stock),
    GREATEST(0, fr.predicted_quantity * COALESCE(pc.restock_cycle_days, 7) + p.safety_stock - p.stock),
    0
FROM product p
JOIN forecast_result fr ON fr.product_id = p.id AND fr.forecast_date = '2026-03-28'
JOIN product_category pc ON p.category_id = pc.id
WHERE p.stock <= p.safety_stock;  -- 红灯条件

-- Step 2: 找出有红灯商品的分类，再找出这些分类下的黄灯商品
-- （实现：有红灯的分类，黄灯商品顺带补货）
-- 具体逻辑在 Java Service 中实现，SQL 较复杂
```

### 5.2 查询进货建议列表

```sql
-- 按分类分组展示
SELECT
    category_name,
    COUNT(*) AS total_count,
    SUM(CASE WHEN light_status = 1 THEN 1 ELSE 0 END) AS red_count,
    SUM(CASE WHEN light_status = 2 THEN 1 ELSE 0 END) AS yellow_count,
    SUM(suggested_quantity) AS total_quantity,
    SUM(suggested_quantity * purchase_price) AS total_amount
FROM purchase_suggestion
WHERE status = 0
GROUP BY category_name
ORDER BY red_count DESC, category_name;
```

### 5.3 生成进货单

```sql
-- Step 1: 创建进货单主表
INSERT INTO purchase_order (order_no, order_date, total_quantity, total_amount, status, operator)
VALUES ('PO-20260328-001', '2026-03-28', 0, 0, 0, 'admin');

-- Step 2: 插入进货单明细
INSERT INTO purchase_order_item (order_id, order_no, product_id, product_code, product_name, category_name, purchase_price, quantity, subtotal_amount, suggestion_id)
SELECT
    LAST_INSERT_ID(),
    'PO-20260328-001',
    ps.product_id,
    ps.product_code,
    ps.product_name,
    ps.category_name,
    ps.purchase_price,
    ps.final_quantity,
    ps.final_quantity * ps.purchase_price,
    ps.id
FROM purchase_suggestion ps
WHERE ps.id IN (1, 2, 3);  -- 用户选择的建议

-- Step 3: 更新进货单汇总
UPDATE purchase_order po
SET
    total_quantity = (SELECT SUM(quantity) FROM purchase_order_item WHERE order_id = po.id),
    total_amount = (SELECT SUM(subtotal_amount) FROM purchase_order_item WHERE order_id = po.id)
WHERE po.id = LAST_INSERT_ID();

-- Step 4: 更新进货建议状态
UPDATE purchase_suggestion
SET status = 1
WHERE id IN (1, 2, 3);
```

## 七、总结

### 7.1 最终表结构

| 表名 | 用途 | 字段数 |
|------|------|--------|
| `product_category` | 新增 restock_cycle_days | 1个新字段 |
| `purchase_suggestion` | 进货建议 | 18个字段 |
| `purchase_order` | 进货单主表 | 11个字段 |
| `purchase_order_item` | 进货单明细 | 12个字段 |

### 7.2 设计亮点

1. **极简主义**：不设计供应商表、批次表，够用就好
2. **单一职责**：预测表只管预测，进货建议表只管进货建议
3. **快照机制**：库存、安全库存记录快照，数据可追溯
4. **红黄绿灯**：light_status 字段支持前端差异化展示
5. **用户可调整**：adjusted_quantity 支持用户手动修改进货量
