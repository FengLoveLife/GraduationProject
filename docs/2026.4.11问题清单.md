# 2026.4.11 问题清单 & 优化方案

## 一、已确认的问题清单

### 问题 1：详情弹窗趋势图是假数据（严重）

**位置**：`Frontend/supermarket/src/views/forecast/predict/index.vue` 第 383-395 行

**现象**：点击商品打开详情弹窗，"近7天销量趋势"图表中的"实际销量"数据是随机数生成的，
每次打开都不同，和真实历史数据毫无关系。

**根本原因**：
```javascript
const actual = [
    baseQty - 5 + Math.floor(Math.random() * 3),  // 纯随机数
    ...
]
```

**修复方案**：调用 `/forecast/compare` 接口，传入 `productId`，获取该商品真实的
历史实际销量 + 历史预测值渲染图表。

---

### 问题 2：feature_engineering.py 中 BASE_DATE 硬编码（严重）

**位置**：`PythonService/forecast/feature_engineering.py`

**现象**：
```python
BASE_DATE = "2026-04-01"    # 死的！
TRAINING_DAYS = 90
```

无论何时运行预测，训练数据永远只取到 4 月 1 日之前的 90 天，
4 月 1 日之后产生的新销售数据完全不会被模型学习到。

**修复方案**：动态从数据库取最新销售日期：
```python
# 从数据库动态获取训练截止日期
result = db.query_one("SELECT MAX(sale_date) as max_date FROM sales_order_item")
BASE_DATE = str(result['max_date'])  # 动态计算，永远使用最新数据
```

---

### 问题 3：多天预测结果表格体验差（中等）

**位置**：`Frontend/supermarket/src/views/forecast/predict/index.vue`

**现象**：选 7 天预测，结果是 101 × 7 = 707 行数据平铺在表格里。
用户想看"乐事薯片未来7天走势"，要在707行里找7条。

**修复方案**：见下方"多天预测设计方案"章节（核心优化点）。

---

### 问题 4：模型精度指标对用户不可见（轻微）

**位置**：前端预测页面

**现象**：Python 服务的 `/forecast/status` 接口已返回 MAE、MAPE、R² 等指标，
但前端完全没有展示，答辩时无法向老师展示模型准确率数据。

**修复方案**：在预测配置卡片下方增加"模型状态"区域，
调用 `/forecast/status` 展示：训练时间、MAE、MAPE、R²。

---

### 问题 5：无法从前端触发模型重训（可选）

**现象**：重新训练模型需要手动跑 Python 脚本，操作繁琐。

**修复方案**：在预测页增加"重新训练模型"按钮，
调用 Python 服务的 `/forecast/train` 接口，前端展示训练进度。

## 二、核心设计问题：多天预测的存储与展示

> 这是困扰最久的设计问题，下面给出完整的思路和最终方案。

### 问题的本质

预测天数（1/3/7）带来三个子问题：

1. 多天数据怎么存？
2. 多天数据怎么展示？
3. 重复预测同一天怎么处理？

---

### 存储层设计（已有，且正确）

`forecast_result` 表有唯一约束：
```sql
UNIQUE KEY uk_product_date (product_id, forecast_date)
```

**结论：存储层以"商品+日期"为原子单位，一个商品一天只有一条预测记录。**

Python 服务使用 UPSERT 写入：
```sql
INSERT INTO forecast_result (...) VALUES (...)
ON DUPLICATE KEY UPDATE predicted_quantity = VALUES(predicted_quantity)
```

**这个设计是完全正确的，不需要改。**

---

### 重复预测问题（已被 UPSERT 自动解决）

**场景举例**：
- 4月11日预测"未来7天"→ 写入 4/12、4/13、...、4/18 共7天
- 4月12日预测"未来3天"→ 写入 4/13、4/14、4/15 共3天

4/13、4/14、4/15 这三天的记录，第二次预测会覆盖第一次预测，
因为 UPSERT 遇到重复 (product_id, forecast_date) 直接 UPDATE。

**结论：不会有重复数据，每个商品每天永远只有一条记录，始终是最新预测。**

---

### 展示层设计（重点优化）

多天预测结果的展示要服务两种业务目标：

#### 目标 A：运营规划（按天看）
- 老板想知道：明天哪些商品卖得多？后天呢？
- 需要：以"某一天"为视角，看全部商品当天的预测销量

#### 目标 B：进货决策（按商品汇总）
- 采购员想知道：这 7 天我需要进多少货？
- 需要：以"某商品"为视角，看它在整个预测周期内的总销量，据此决定进货量

**这两个目标需要两种展示形态，不是非此即彼的关系。**

---

### 最终方案：双视图设计

#### 视图 1：按日期浏览（运营视图）

在结果表格上方加一排日期标签（Tab 或 Radio Button），
默认显示第一天，点击切换查看每一天的数据。

```
[2026-04-12] [2026-04-13] [2026-04-14] ... [2026-04-18]
     ↑当前选中

商品名称    | 分类 | 当天预测销量 | 当前库存 | 状态
乐事薯片    | 零食 |    28 件     |   145   | 库存充足
农夫山泉    | 饮料 |    67 件     |    30   | 需要补货
...
```

数据量：101 行，清晰易读。

#### 视图 2：按商品汇总（进货视图）

每行是一个商品，展示整个预测周期内的汇总数据。

```
商品名称 | 分类 | 预测周期总销量 | 当前库存 | 建议进货量 | 状态
乐事薯片 | 零食 |     196 件     |   145   |    +71    | 需补货
农夫山泉 | 饮料 |     469 件     |    30   |   +449    | 必须补货
...
```

**建议进货量计算公式（周期版）**：
```
建议进货量 = 预测周期总销量 + 安全库存 - 当前库存
```

这比原来的单日进货建议更合理，因为采购一次是为了覆盖整个补货周期。

---

### 详情弹窗

点击任意商品，弹窗展示：
- 该商品在预测周期内每一天的预测销量（折线图）
- 历史近7天实际销量（真实数据，调用 compare 接口）
- 库存分析
