### 2026-03-29 下午更新（三）- Top-Down架构重构

**发现问题：Bottom-Up架构的"零和博弈"**

根据《中间问答操作.md》的分析，原Bottom-Up架构存在致命缺陷：
- 固定订单数 → 商品用权重"抢"名额
- 炎热天气可乐权重暴增 → 其他商品销量暴跌
- 各商品销量互相踩踏，破坏独立波动规律

**解决方案：Top-Down架构（顶层向下生成）**

重构 `sales_generator.py`，采用正确的生成流程：
1. **计算目标清单**：遍历101商品，严格按公式计算应售数量
2. **构建商品池**：把所有商品放入池子并打乱
3. **打包订单**：从池子抓取商品组成订单，直到池子清空

**核心优势：**
- 各商品销量独立波动，不受其他商品影响
- 炎热天气 → 整体客流增加 → 订单数自然增加
- 预测模型能精准捕捉每个影响因子

### 2026-03-29 下午更新（五）- 两大优化建议采纳

**多AI交叉验证结果：**

| AI | 问题 | 建议 |
|-----|------|------|
| AI1 | 大数定律抵消效应：101个商品独立波动加总时正负抵消，导致总营业额曲线过于平滑 | 引入全局大盘波动（±5%）+ 单品波动 |
| AI2 | 小销量商品（base=4）使用round()四舍五入，导致销量永远是固定值 | 概率进位法：3.2有20%概率变成4，80%是3 |

**采纳决策：**
- AI2建议（概率进位法）：**必须采纳**，直接影响LightGBM模型训练效果
- AI1建议（全局大盘波动）：采纳，答辩时展示总营业额图表不会被内行看出"假数据"痕迹

**代码修改：**
```python
# 优化1：全局大盘波动
global_noise = random.uniform(-0.05, 0.05)  # 每天±5%

# 优化2：单品波动调小（因为要加全局波动）
product_noise_limit = config.get('noise_range', 0.15) - 0.05
local_noise = random.uniform(-product_noise_limit, product_noise_limit)

# 最终综合波动 = 全局大盘波动 + 单品自身波动
total_noise = global_noise + local_noise

# 优化3：概率进位法
int_part = int(exact_target)          # 整数部分
frac_part = exact_target - int_part   # 小数部分
target = int_part + 1 if random.random() < frac_part else int_part
```

**优化验证结果：**
| 优化 | 验证结果 |
|------|----------|
| 全局大盘波动 | 工作日总营业额波动范围24.4%，标准差1,174元 ✓ |
| 概率进位法 | SP3103日均4.7,范围[3-7],波动84.8%；SP3106日均4.0,范围[3-5],波动50% ✓ |
