## 一、什么是LightGBM？

LightGBM的全称是 **Light Gradient Boosting Machine**，翻译过来是"轻量级梯度提升机"。它是微软在2017年开源的一个机器学习框架，专门用于处理表格数据的预测问题。

简单理解：**它是一个"决策树集成"模型，通过组合很多棵决策树来做出预测。**

举个例子，假设我要预测可乐明天的销量：

```
决策树1判断：如果是周末，销量+20%
决策树2判断：如果是晴天，销量+5%
决策树3判断：如果昨天卖得好，今天大概率也不错
...
最终预测 = 所有决策树预测结果的加权平均
```

每一棵树都在"纠正"前面树的错误，最终组合起来形成一个强大的预测模型。

## 二、为什么选择LightGBM进行销量预测？

在选择LightGBM之前，我调研了很多方案，下面详细说明选择理由：

### 2.1 为什么不选传统统计方法？

**ARIMA（自回归积分滑动平均模型）**

这是教科书上经典的时间序列方法。但有一个致命问题：**只能处理单变量**。

什么意思呢？ARIMA只能根据过去的销量预测未来销量，没法把天气、节假日、促销这些因素考虑进去。但实际超市销售很明显受这些因素影响——周末销量就是比平时高，春节那几天更是翻倍。所以ARIMA不太适合我的场景。

**Prophet（Facebook开源）**

这个对节假日处理得很好，但它更适合做**趋势预测**，比如预测明年整年的销售趋势。我要做的是**商品级别的日销量预测**，每天每个商品都要给一个具体数字，Prophet不太适合这种场景。

### 2.2 为什么不选其他机器学习方法？

**XGBoost**

XGBoost和LightGBM都是梯度提升决策树，原理类似。但我最终没选XGBoost，原因如下：

| 对比项 | XGBoost | LightGBM |
|--------|---------|----------|
| 训练速度 | 基准 | 快3-10倍 |
| 内存占用 | 较大 | 较小 |
| 类别特征处理 | 需要One-Hot编码 | 原生支持 |

最后一点很关键。我的数据里有大量类别特征：商品ID、天气类型、节假日类型。用XGBoost的话，需要先把这些转成One-Hot编码（比如天气有4种，就要转成4列0和1），代码会变得很复杂。LightGBM可以直接处理，代码简洁很多。

**LSTM（长短期记忆网络）**

这是深度学习的方法，理论上能学到更复杂的模式。但有几个问题：
- 学习曲线陡峭，我Python基础还在学习中
- 需要大量数据，我目前只有30天数据，容易过拟合
- 训练时间长，调试成本高

对于毕业设计来说，性价比不高。

### 2.3 选择LightGBM的核心原因

**原因一：类别特征原生支持**

我的特征结构：

```
商品特征：product_id（类别）、category_id（类别）
时间特征：day_of_week（类别1-7）、month（类别1-12）
日历因子：weather（类别：晴/多云/雨/雪）、holiday_name（类别）
二值特征：is_weekend、is_holiday
```

这些都是类别特征。LightGBM可以直接处理，不需要额外编码：

```python
# LightGBM代码示例
model.fit(
    X_train, y_train,
    categorical_feature=['product_id', 'category_id', 'day_of_week', 'weather']
)
```

**原因二：训练速度快**

LightGBM采用了一种叫"直方图算法"的技术，把连续值离散化成直方图，减少了计算量。实测在相同数据上，比XGBoost快3-10倍。

这意味着我可以快速尝试不同的特征组合、调整参数，加快开发迭代。

**原因三：内存占用小**

对于我这个在个人电脑上开发的项目来说，内存占用很重要。LightGBM比XGBoost节省约50%内存。

**原因四：业界成熟**

- 微软开源，质量有保证
- Kaggle竞赛中使用率很高，是结构化数据预测的主流选择
- 文档完善，中文教程也很多

## 三、如何考虑影响因子？

这是预测模型能否准确的关键。我设计的特征体系如下：

### 3.1 影响因子分类

```
┌─────────────────────────────────────────────────────────┐
│                    销量影响因子                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  【时间规律】           【日历因子】                      │
│  ├── 星期几             ├── 是否周末                      │
│  ├── 月份               ├── 是否节假日                    │
│  ├── 是否月初/月末       ├── 节假日名称                   │
│                         └── 天气情况                      │
│  【历史销量】                                            │
│  ├── 昨天销量           【商品特征】                      │
│  ├── 上周同一天销量     ├── 商品ID                        │
│  ├── 近7天平均销量      └── 商品分类                      │
│  └── 近7天销量波动                                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 各因子的作用机制

**时间规律因子**

超市销售有明显的时间规律：
- 周六周日客流大，销量高
- 月初发工资后消费多，月末相对少
- 春节、国庆等节假日前夕销量暴增

模型会从历史数据中自动学习这些规律。比如它会发现"周六的销量通常是工作日的1.3倍"，然后把这条规则应用到预测中。

**天气因子**

天气对客流的影响：
- 晴天：正常
- 多云：正常
- 小雨：客流减少，销量下降约15%
- 大雨：客流大幅减少，销量下降约40%
- 炎热：饮料类商品销量上升

**节假日因子**

节假日对销售的影响：
- 普通节假日：销量提升20%-50%
- 春节前夕：销量可能翻倍
- 特殊节日（如情人节）：特定商品（鲜花、巧克力）销量暴增

**历史销量因子**

这是最重要的特征。商品销量有"惯性"：
- 昨天卖得好，今天大概率也不错
- 上周同一天的销售情况有参考价值
- 近期销量平均值代表当前销售水平

模型会把历史销量作为基准，再根据时间、天气、节假日等因素进行调整。

### 3.3 特征工程实现

把原始数据转换成模型能理解的特征：

```python
def create_features(sales_data, calendar_data):
    """创建预测特征"""

    features = []

    for record in sales_data:
        date = record['date']
        product_id = record['product_id']

        # 时间特征
        feature = {
            'day_of_week': date.weekday() + 1,      # 1-7
            'month': date.month,                      # 1-12
            'is_weekend': 1 if date.weekday() >= 5 else 0,
        }

        # 合并日历因子（从calendar_factor表获取）
        calendar = get_calendar_factor(date)
        feature['is_holiday'] = calendar['is_holiday']
        feature['holiday_name'] = calendar['holiday_name']
        feature['weather'] = calendar['weather']

        # 计算历史销量特征
        feature['lag_1'] = get_yesterday_sales(product_id, date)      # 昨天销量
        feature['lag_7'] = get_last_week_sales(product_id, date)      # 上周同日
        feature['rolling_7_mean'] = get_7day_avg(product_id, date)    # 7天均值

        features.append(feature)

    return features
```

## 四、LightGBM是怎样的算法？

### 4.1 基本原理（通俗版）

想象一下，你要判断明天可乐能卖多少瓶。LightGBM的工作方式就像一个"问答游戏"：

```
问题1：明天是周末吗？
  └─ 是 → 大概能卖50瓶左右
  └─ 否 → 继续

问题2：明天天气怎么样？
  └─ 晴天 → 销量+10%
  └─ 雨天 → 销量-15%
  └─ 多云 → 销量不变

问题3：昨天卖了多少？
  └─ 卖了40瓶以上 → 今天应该也不错
  └─ 卖得少 → 今天可能也一般

...经过很多个问题后...

最终预测：明天大概卖45瓶
```

这就是决策树的工作方式：不断"提问-分支"，最后给出预测值。

LightGBM用的是"梯度提升"的方法，意思是：

1. 先训练一棵树，看看预测误差
2. 再训练第二棵树，专门纠正第一棵树的错误
3. 再训练第三棵树，纠正前两棵树的剩余错误
4. ...如此循环，训练很多棵树

最终预测 = 所有树预测结果的加权求和

### 4.2 LightGBM的技术创新

LightGBM相比传统梯度提升树，有两个重要创新：

**创新一：直方图算法（Histogram Algorithm）**

传统方法每次分裂都要遍历所有数据，计算量大。LightGBM把连续值离散化成直方图（比如把价格分成若干个区间），减少了计算量。

**创新二：按叶子生长（Leaf-wise）**

传统方法是"按层生长"，每一层所有节点都分裂。LightGBM是"按叶子生长"，每次只分裂误差最大的那个叶子。这样能更快地降低误差，用更少的树达到相同效果。

```
按层生长（传统）：          按叶子生长（LightGBM）：
      根                          根
     /  \                        /  \
    A    B                      A    B
   / \  / \                    / \
  C  D E  F                   C   D
 (所有节点都分裂)              (只分裂误差大的节点)
```

### 4.3 训练过程示意

```python
import lightgbm as lgb

# 1. 准备数据
train_data = lgb.Dataset(X_train, label=y_train)

# 2. 设置参数
params = {
    'objective': 'regression',    # 回归任务（预测数值）
    'metric': 'mae',              # 评估指标：平均绝对误差
    'num_leaves': 31,             # 叶子节点数
    'learning_rate': 0.1,         # 学习率
    'n_estimators': 100,          # 树的数量
}

# 3. 训练模型
model = lgb.train(params, train_data, num_boost_round=100)

# 4. 预测
predictions = model.predict(X_test)
```

核心代码就这几行，LightGBM的API设计得很简洁。

### 7.2 为什么用两个服务？

可能有人会问：为什么不直接在Java里做预测？

主要考虑几点：

1. **技术生态**：机器学习领域的工具链基本都在Python生态里。LightGBM、Pandas、NumPy这些库非常成熟，Java虽然有调用方式，但用起来麻烦很多。

2. **开发效率**：用Python写特征工程和模型训练，几十行代码就能搞定。用Java写同样的功能，代码量可能翻倍。

3. **解耦**：预测服务独立部署，后面如果需要优化模型、调整参数，只需要重新训练模型文件，不影响Java后端的业务逻辑。

4. **可扩展**：以后如果模型需要升级（比如换成深度学习模型），只需要改Python服务，前端和Java后端不用动。

## 八、进货建议算法

预测出销量后，怎么生成进货建议？这个算法设计得相对简单但实用。

### 8.1 核心公式

```
建议进货量 = 预测销量 + 安全库存 - 当前库存
```

**解释**：
- 预测销量：模型预测的明天销量
- 安全库存：为了应对突发情况的缓冲库存（用户可以设置）
- 当前库存：现在仓库里有多少

如果计算结果是正数，说明需要进货；如果是负数或零，说明库存充足。

### 8.2 示例

假设可口可乐：
- 预测明天销量：45瓶
- 安全库存：20瓶
- 当前库存：30瓶

计算：45 + 20 - 30 = 35瓶

系统会建议进货35瓶。

### 8.3 进货建议列表

系统会自动筛选出所有"建议进货量 > 0"的商品，生成进货建议清单，用户可以：
- 调整进货数量
- 移除不需要进货的商品
- 一键生成进货单
