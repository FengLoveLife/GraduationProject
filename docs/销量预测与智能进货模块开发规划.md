## 一、为什么现在才开始做预测模块？

这个问题我需要先说明一下。

说实话，预测模块是整个系统最核心也是最复杂的部分。我一开始就清楚这一点，所以选择了"先易后难"的开发策略：

1. **先把基础业务模块做好**：商品管理、销售数据导入、统计分析这些是预测模块的数据基础，没有这些就没有历史数据，预测模型就训练不了。

2. **预留好数据接口**：比如"日历因子管理"这个模块，很多同学可能觉得它和销售关系不大，但实际上它是预测模型的重要特征来源——天气、节假日、周末这些因素对销量影响很大。

3. **学习曲线问题**：我之前主要用Java，Python接触不多。而预测模块需要用到Python的机器学习库（LightGBM、Pandas等），我需要边学边做。这段时间我一直在补Python和机器学习的基础知识。

现在基础模块都完成了，数据结构也设计好了，接下来就是专心攻克预测模块了。

## 二、技术选型：为什么选择LightGBM？

这个问题我调研了很久，也走过一些弯路。

### 2.1 我考虑过的方案

最开始我其实想用ARIMA模型，因为课本上讲过，比较熟悉。但后来发现ARIMA有个大问题：**只能处理单变量**。也就是说，它只能根据过去的销量预测未来销量，没法把天气、节假日这些因素考虑进去。但实际超市销售很明显受这些因素影响——周末销量就是比平时高，春节那几天更是翻倍。所以ARIMA不太合适。

然后我看了Prophet，Facebook开源的。这个对节假日处理得很好，但问题是它更适合做**趋势预测**，比如预测明年整年的销售趋势。我要做的是**商品级别的日销量预测**，每天每个商品都要给一个具体数字，Prophet不太适合这种场景。

后来我看了XGBoost和LightGBM这两个机器学习模型。这两个都是决策树集成模型，在Kaggle竞赛里用得特别多。我对比了一下：

| 对比项 | XGBoost | LightGBM |
|--------|---------|----------|
| 训练速度 | 基准 | 快3-10倍 |
| 内存占用 | 较大 | 较小 |
| 类别特征处理 | 需要One-Hot编码 | 原生支持 |

最后一点很关键：**LightGBM原生支持类别特征**。什么意思呢？比如商品ID、天气类型、节假日类型这些，都是类别型的数据。用XGBoost的话，我得先把它们转成One-Hot编码，代码会变得很复杂。LightGBM可以直接处理，代码简洁很多。

综合考虑下来，我选了LightGBM。

### 2.2 模型能学到什么？

这个问题挺有意思的。LightGBM本质上是决策树的集成，它的工作方式是不断"提问-分支"，最后给出预测值。

对于销量预测，模型会学到类似这样的规律：

- 如果是周六或周日，销量大概会增加30%
- 如果是春节假期，销量大概会增加50%
- 如果天气是雨天，销量可能会下降10%
- 如果昨天卖得很好，今天大概率也不错
- 如果上周同一天卖得好，这周同一天也会不错

这些都是从历史数据里自动学到的规律，不需要我人工去设定规则。这就是机器学习的好处。

## 三、预测需要用到哪些数据？

### 3.1 已有的数据基础

目前系统已经积累了以下数据：

| 数据类型 | 数据表 | 数据量 |
|----------|--------|--------|
| 商品信息 | product | 约200条 |
| 销售订单 | sales_order + sales_order_item | 约30天，每天20-50单 |
| 日历因子 | calendar_factor | 每天一条，包含节假日、天气 |

### 3.2 特征工程设计

这是预测模块的关键。我需要把原始数据转换成模型能理解的特征。

**时间特征**（从日期直接提取）：
- 星期几（1-7）
- 月份（1-12）
- 是否月初/月末
- 是否周末

**日历因子**（从calendar_factor表获取）：
- 是否节假日
- 节假日名称（春节、国庆等）
- 天气情况（晴、雨、雪）

**历史销量特征**（需要计算）：
- 昨天销量
- 上周同一天销量
- 近7天平均销量
- 近7天销量标准差（反映销量波动程度）
- 近14天平均销量

**商品特征**：
- 商品ID（类别特征，LightGBM可以直接处理）
- 商品分类ID

### 3.3 特征怎么帮助预测？

举个例子：

假设我要预测"可口可乐500ml"在3月22日（周六）的销量。

我需要告诉模型：
- 这天是周六（is_weekend = 1）
- 不是节假日（is_holiday = 0）
- 天气晴（weather = 0）
- 昨天卖了45瓶（lag_1 = 45）
- 上周六卖了52瓶（lag_7 = 52）
- 近7天平均每天卖48瓶（rolling_7_mean = 48）

模型会根据这些特征，结合它从历史数据学到的规律，给出一个预测值。

## 四、系统架构设计

### 4.1 整体架构

我采用的是**前后端分离 + 微服务**的架构：

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端 (Vue 3)                              │
│   预测结果页面        进货建议页面        预测趋势图表            │
└─────────────────────────────────────────────────────────────────┘
                              │ HTTP请求
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Java后端 (Spring Boot)                        │
│   ForecastController                                             │
│     - POST /api/forecast/predict      单商品预测                │
│     - POST /api/forecast/batch        批量预测                  │
│     - GET  /api/forecast/suggestion   进货建议                  │
│                                                                  │
│   ForecastService                                                │
│     - 计算历史特征（从MySQL获取销售数据）                        │
│     - 调用Python预测服务                                         │
│     - 计算进货建议                                               │
└─────────────────────────────────────────────────────────────────┘
                              │ HTTP调用
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Python预测服务 (FastAPI)                       │
│   /predict           单商品预测接口                              │
│   /predict/batch     批量预测接口                                │
│   /health            健康检查接口                                │
│                                                                  │
│   加载训练好的LightGBM模型                                       │
│   接收特征 → 模型推理 → 返回预测值                               │
└─────────────────────────────────────────────────────────────────┘
                              ▲
                              │ 离线训练
┌─────────────────────────────────────────────────────────────────┐
│                    模型训练脚本 (离线执行)                        │
│   1. 从MySQL读取历史销售数据                                     │
│   2. 数据清洗、特征工程                                          │
│   3. 划分训练集/验证集/测试集                                    │
│   4. 训练LightGBM模型                                           │
│   5. 模型评估（MAE、MAPE等指标）                                │
│   6. 保存模型文件                                                │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 为什么用两个服务？

可能有人会问：为什么不直接在Java里做预测？

主要考虑几点：

1. **技术生态**：机器学习领域的工具链基本都在Python生态里。LightGBM、Pandas、NumPy这些库非常成熟，Java虽然有调用方式，但用起来麻烦很多。

2. **开发效率**：用Python写特征工程和模型训练，几十行代码就能搞定。用Java写同样的功能，代码量可能翻倍。

3. **解耦**：预测服务独立部署，后面如果需要优化模型、调整参数，只需要重新训练模型文件，不影响Java后端的业务逻辑。

4. **可扩展**：以后如果模型需要升级（比如换成深度学习模型），只需要改Python服务，前端和Java后端不用动。

## 五、进货建议算法

预测出销量后，怎么生成进货建议？这个算法设计得相对简单但实用。

### 5.1 核心公式

```
建议进货量 = 预测销量 + 安全库存 - 当前库存
```

**解释**：
- 预测销量：模型预测的明天销量
- 安全库存：为了应对突发情况的缓冲库存（用户可以设置）
- 当前库存：现在仓库里有多少

如果计算结果是正数，说明需要进货；如果是负数或零，说明库存充足。

### 5.2 示例

假设可口可乐：
- 预测明天销量：45瓶
- 安全库存：20瓶
- 当前库存：30瓶

计算：45 + 20 - 30 = 35瓶

系统会建议进货35瓶。

### 5.3 进货建议列表

系统会自动筛选出所有"建议进货量 > 0"的商品，生成进货建议清单，用户可以：
- 调整进货数量
- 移除不需要进货的商品
- 一键生成进货单

## 七、技术难点与应对

### 7.1 数据量不足的问题

**问题**：目前只有30天的模拟数据，对于机器学习来说偏少。

**应对**：
1. 使用交叉验证充分利用数据
2. 模型参数设置保守一些，避免过拟合
3. 如果预测效果不好，可以加入规则策略作为兜底（比如用近7天平均值）

### 7.2 新商品冷启动问题

**问题**：新上架的商品没有历史数据，无法预测。

**应对**：
1. 对于没有历史数据的商品，使用分类平均值作为预测
2. 后续可以增加"相似商品"推荐逻辑

### 7.3 Python服务稳定性

**问题**：Python服务挂了怎么办？

**应对**：
1. Java后端增加健康检查接口，检测Python服务状态
2. 如果Python服务不可用，返回提示信息，前端显示"预测服务暂不可用"
3. 考虑增加简单的规则预测作为降级方案

## 八、预期效果

### 8.1 功能目标

1. 用户可以选择预测日期，查看所有商品的预测销量
2. 系统自动筛选需要进货的商品，生成进货建议
3. 支持一键生成进货单

### 8.2 预测准确率目标

考虑到数据量和业务场景，我设定的目标是：
- MAPE（平均绝对百分比误差）控制在 **25% 以内**
- 能够正确反映周末、节假日对销量的影响趋势

这个目标对毕业设计来说是合理的，实际应用中可以通过增加数据量、优化特征来提升准确率。
