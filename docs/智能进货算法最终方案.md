# 智能进货算法最终方案

> 文档目的：确定智能进货模块的核心算法逻辑
> 创建时间：2026年3月24日
> 状态：已确认

## 一、核心问题

### 问题1：不同商品的进货周期不同

生鲜保质期短，需要快进快出；日用品保质期长，可以一个月进一次。如何智能适配？

### 问题2：如何实现分类联合补货

同一分类的商品，进货周期相同。当一个商品库存紧张时，是否应该把同分类的其他商品一起补货？

**子问题：**
- 多少个商品出现库存问题时触发进货？
- 如果只是个别商品促销导致库存紧张，要不要拉着全分类一起补？
- 顺带补货时，是补全部商品，还是有选择地补？

---

## 二、解决方案

### 2.1 品类补货周期

将"补货周期"配置在商品分类表上，不同分类设置不同周期：

| 分类 | 补货周期(天) | 原因 |
|------|-------------|------|
| 生鲜果蔬 | 3 | 保质期短，快进快出 |
| 饮料 | 14 | 销量稳定 |
| 休闲零食 | 14 | 销量稳定 |
| 粮油调味 | 30 | 销量慢 |
| 日用百货 | 30 | 保质期长 |

### 2.2 双水位模型

引入两个核心概念：

| 概念 | 含义 | 作用 |
|------|------|------|
| **安全库存(MIN)** | 库存警戒线 | 触发进货报警 |
| **目标库存(MAX)** | 进货目标水位 | 决定进货到多少 |

**计算公式：**
```
目标库存 = 日均销量 × 补货周期 + 安全库存
建议进货量 = 目标库存 - 当前库存
```

**日均销量来源：**
- 优先使用AI预测值（预测模块就绪后）
- 降级使用近7天平均销量

### 2.3 红黄绿灯·动态凑单模型

通过数学公式评估每个商品的"健康度"，决定是否需要补货：

```
┌─────────────────────────────────────────────────────────────┐
│                    红黄绿灯判断规则                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🔴 红灯（必须进货）                                         │
│     条件：当前库存 <= 安全库存                               │
│     动作：发起进货，是这次补货的"带头大哥"                    │
│                                                             │
│  🟡 黄灯（顺带进货）                                         │
│     条件：安全库存 < 当前库存 <= 安全库存 + 周期销量×30%      │
│     含义：还能撑周期的30%，既然红灯要进货，就顺带一起补        │
│     动作：加入进货列表                                       │
│                                                             │
│  🟢 绿灯（不进货）                                           │
│     条件：当前库存 > 安全库存 + 周期销量×30%                  │
│     含义：库存健康，即使同分类有红灯也不动                    │
│     动作：跳过，不进货                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**黄灯线公式说明：**
```
黄灯线 = 安全库存 + 周期预测销量 × 30%

举例：
- 安全库存 = 10瓶
- 周期销量 = 50瓶（30天的总销量）
- 黄灯线 = 10 + 50 × 30% = 25瓶

含义：当前库存低于25瓶时，说明库存撑不过周期的30%，应该顺带补货
```

## 三、场景验证

### 场景1：个别商品促销爆单

**背景：** 5种洗发水，周期30天，刚进货10天。其中A因促销卖爆了，其他4个库存充足。

**数据：**
| 商品 | 目标库存 | 安全库存 | 当前库存 | 黄灯线 | 状态 |
|------|---------|---------|---------|--------|------|
| A（促销爆款） | 60 | 10 | 8 | 25 | 🔴红灯 |
| B | 60 | 10 | 40 | 25 | 🟢绿灯 |
| C | 60 | 10 | 42 | 25 | 🟢绿灯 |
| D | 60 | 10 | 38 | 25 | 🟢绿灯 |
| E | 60 | 10 | 45 | 25 | 🟢绿灯 |

**结果：** 只补A，其他商品库存健康不动。✅ 避免了促销误判导致的盲目全补

### 场景2：平稳销售，部分商品快卖完

**背景：** 5种洗发水，周期30天，已卖25天。A卖完了，B也快了。

**数据：**
| 商品 | 目标库存 | 安全库存 | 当前库存 | 黄灯线 | 状态 |
|------|---------|---------|---------|--------|------|
| A | 60 | 10 | 8 | 25 | 🔴红灯 |
| B | 60 | 10 | 18 | 25 | 🟡黄灯 |
| C | 60 | 10 | 35 | 25 | 🟢绿灯 |
| D | 60 | 10 | 40 | 25 | 🟢绿灯 |
| E | 60 | 10 | 38 | 25 | 🟢绿灯 |

**结果：** A和B一起补货，C、D、E不动。✅ 实现了智能凑单

**进货量计算：**
- A：60 - 8 = 52瓶
- B：60 - 18 = 42瓶
- 合计：94瓶

## 四、完整流程

```
┌─────────────────────────────────────────────────────────────┐
│                    智能进货流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 扫描红灯商品                                       │
│  ───────────────────                                        │
│  遍历所有商品，找出 当前库存 <= 安全库存 的商品              │
│  按商品分类分组                                             │
│                                                             │
│  Step 2: 按分类检查黄灯商品                                 │
│  ─────────────────────────                                  │
│  对于有红灯的分类：                                          │
│    - 查询该分类下所有商品                                    │
│    - 计算每个商品的状态（红/黄/绿）                          │
│    - 筛选出红灯和黄灯商品                                    │
│                                                             │
│  Step 3: 计算进货建议                                       │
│  ───────────────────                                        │
│  对于每个需要补货的商品：                                    │
│    - 目标库存 = 日均销量 × 补货周期 + 安全库存               │
│    - 建议进货量 = 目标库存 - 当前库存                        │
│                                                             │
│  Step 4: 生成进货建议单                                     │
│  ───────────────────                                        │
│  按分类分组展示进货建议                                      │
│  用户确认后生成进货单                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 五、数据库设计

### 5.1 修改分类表

```sql
ALTER TABLE `product_category`
ADD COLUMN `restock_cycle_days` INT DEFAULT 7 COMMENT '补货周期(天)';
```

### 5.2 新建进货建议表

```sql
CREATE TABLE `purchase_suggestion` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 关联信息
    `product_id` BIGINT NOT NULL COMMENT '商品ID',
    `category_id` BIGINT NOT NULL COMMENT '分类ID',

    -- 冗余字段
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称',
    `category_name` VARCHAR(50) NOT NULL COMMENT '分类名称',
    `purchase_price` DECIMAL(10,2) DEFAULT NULL COMMENT '进货单价',

    -- 库存状态
    `current_stock` INT NOT NULL COMMENT '当前库存',
    `safety_stock` INT NOT NULL COMMENT '安全库存',
    `daily_sales` DECIMAL(10,2) DEFAULT 0 COMMENT '日均销量',
    `target_stock` INT NOT NULL COMMENT '目标库存',

    -- 灯位状态
    `light_status` TINYINT DEFAULT 1 COMMENT '灯位: 1红灯 2黄灯',

    -- 进货建议
    `suggested_quantity` INT NOT NULL COMMENT '建议进货量',
    `adjusted_quantity` INT DEFAULT NULL COMMENT '调整后进货量',
    `final_quantity` INT NOT NULL COMMENT '最终进货量',

    -- 状态
    `status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态: 0待处理 1已生成进货单 2已忽略',
    `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注',

    -- 审计
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    PRIMARY KEY (`id`),
    KEY `idx_category_id` (`category_id`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='进货建议表';
```

### 5.3 新建进货单主表

```sql
CREATE TABLE `purchase_order` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `order_no` VARCHAR(32) NOT NULL COMMENT '进货单号',

    -- 订单信息
    `total_quantity` INT NOT NULL DEFAULT 0 COMMENT '商品总数量',
    `total_amount` DECIMAL(12,2) NOT NULL DEFAULT 0.00 COMMENT '进货总金额',

    -- 时间信息
    `order_date` DATE NOT NULL COMMENT '下单日期',
    `expected_date` DATE DEFAULT NULL COMMENT '预计到货日期',
    `actual_arrival_date` DATE DEFAULT NULL COMMENT '实际到货日期',

    -- 状态
    `status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态: 0待确认 1已下单 2已完成 3已取消',

    `remark` VARCHAR(500) DEFAULT NULL COMMENT '备注',
    `operator` VARCHAR(50) DEFAULT NULL COMMENT '操作人',

    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_order_no` (`order_no`),
    KEY `idx_order_date` (`order_date`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='进货单主表';
```

### 5.4 新建进货单明细表

```sql
CREATE TABLE `purchase_order_item` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `order_id` BIGINT NOT NULL COMMENT '进货单ID',
    `order_no` VARCHAR(32) NOT NULL COMMENT '进货单号',

    -- 商品信息
    `product_id` BIGINT NOT NULL COMMENT '商品ID',

    -- 冗余字段
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称',
    `category_name` VARCHAR(50) DEFAULT NULL COMMENT '分类名称',

    -- 进货信息
    `purchase_price` DECIMAL(10,2) NOT NULL COMMENT '进货单价',
    `quantity` INT NOT NULL COMMENT '进货数量',
    `subtotal_amount` DECIMAL(12,2) NOT NULL COMMENT '小计金额',

    -- 关联进货建议
    `suggestion_id` BIGINT DEFAULT NULL COMMENT '关联的进货建议ID',

    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

    PRIMARY KEY (`id`),
    KEY `idx_order_id` (`order_id`),
    KEY `idx_product_id` (`product_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='进货单明细表';
```

## 七、答辩话术

> "我的系统采用了**双水位模型与红黄绿灯动态凑单机制**。
>
> 首先，系统根据商品分类自动适配不同的补货周期，生鲜只补3天的量，日用品补30天的量，保证资金周转效率。
>
> 其次，当某商品库存跌破安全库存时，系统不会盲目地把全分类商品都补货，而是通过红黄绿灯模型智能判断：
> - 红灯商品必须补货
> - 黄灯商品（库存撑不过周期30%的）顺带补货
> - 绿灯商品不动
>
> 这样既避免了促销爆款导致的盲目全补，也实现了同分类商品的智能凑单，极大降低了采购频次和物流成本。"
