## 一、现有数据基础

在设计新表之前，先梳理已有的数据表及其在预测模块中的作用：

| 已有表 | 与预测模块的关系 |
|--------|------------------|
| `product` | 提供商品信息、当前库存、安全库存 |
| `product_category` | 提供商品分类信息，可作为预测特征 |
| `sales_order` | 提供历史销售数据，用于训练模型 |
| `sales_order_item` | 提供商品级别的历史销量明细 |
| `calendar_factor` | 提供日期特征（天气、节假日、周末） |
| `inventory_log` | 可用于验证库存变动 |

**结论**：已有的数据表足够支撑预测模型训练，无需额外添加训练数据相关的表。

## 二、冗余字段设计原则

### 2.1 为什么要冗余？

在展示预测结果和进货建议时，需要关联多张表：

```
预测结果列表展示需要的信息：
┌─────────────────────────────────────────────────────────────┐
│ 预测日期 │ 商品名称 │ 商品编码 │ 分类 │ 预测销量 │ 当前库存 │ 安全库存 │
├─────────────────────────────────────────────────────────────┤
│ 03-23   │ 可口可乐 │ SP001   │ 饮料 │   45    │   30    │   20    │
└─────────────────────────────────────────────────────────────┘
         ↑                    ↑              ↑
    forecast_result       product         product_category
```

**不冗余**：需要 JOIN 3张表 (forecast_result + product + product_category)

**有冗余**：单表查询即可，性能更优

### 2.2 冗余字段选择原则

参考现有的 `sales_order_item` 表设计（已有冗余：product_code, product_name, category_id, category_name），我们遵循以下原则：

| 原则 | 说明 |
|------|------|
| **高频查询字段** | 列表展示经常用到的字段考虑冗余 |
| **相对稳定字段** | 商品名称、分类名称等不常变更的字段适合冗余 |
| **快照性质字段** | 记录生成时的状态，如当时的库存、进货价 |
| **避免冗余** | 经常变更的字段不适合冗余（如实时库存，那是快照） |

### 2.3 冗余字段对比分析

| 表 | 无冗余时JOIN数 | 有冗余后 | 性能提升 |
|----|---------------|----------|----------|
| `forecast_result` | 3张表 | 0张表 | ⬆️ 显著 |
| `purchase_suggestion` | 4张表 | 0张表 | ⬆️⬆️ 非常显著 |
| `purchase_order_item` | 2张表 | 0张表 | ⬆️ 显著 |

## 三、新增表设计（含冗余字段）

### 3.1 预测结果表 `forecast_result`

用于存储每次预测的结果，方便历史查询和模型评估。

```sql
-- 预测结果表
CREATE TABLE `forecast_result` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 预测目标
    `forecast_date` DATE NOT NULL COMMENT '预测的目标日期（哪一天的销量）',
    `product_id` BIGINT NOT NULL COMMENT '商品ID',

    -- ========== 冗余字段（减少JOIN查询） ==========
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码(冗余)',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称(冗余)',
    `category_id` BIGINT DEFAULT NULL COMMENT '分类ID(冗余)',
    `category_name` VARCHAR(50) DEFAULT NULL COMMENT '分类名称(冗余)',

    -- 预测输入特征（记录预测时的环境，便于追溯）
    `weather` VARCHAR(20) DEFAULT NULL COMMENT '预测时使用的天气条件',
    `is_holiday` TINYINT DEFAULT 0 COMMENT '预测时是否节假日: 0否 1是',
    `is_weekend` TINYINT DEFAULT 0 COMMENT '预测时是否周末: 0否 1是',

    -- 预测结果
    `predicted_quantity` INT NOT NULL COMMENT '预测销量',
    `confidence` INT DEFAULT NULL COMMENT '预测置信度(0-100)，可选字段',

    -- 实际对比（用于后续评估模型效果）
    `actual_quantity` INT DEFAULT NULL COMMENT '实际销量（预测日期过后填入）',
    `error_rate` DECIMAL(5,2) DEFAULT NULL COMMENT '预测误差率(%)，实际销量确认后计算',

    -- 审计字段
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_forecast_product_date` (`forecast_date`, `product_id`) COMMENT '同一商品同一天只能有一条预测记录',
    KEY `idx_forecast_date` (`forecast_date`) COMMENT '按预测日期查询',
    KEY `idx_product_id` (`product_id`) COMMENT '按商品查询历史预测',
    KEY `idx_category_id` (`category_id`) COMMENT '按分类查询'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='销量预测结果表';
```

**冗余字段说明**：

| 冗余字段 | 来源表 | 理由 |
|----------|--------|------|
| `product_code` | product | 列表展示必需，变更频率极低 |
| `product_name` | product | 列表展示必需，变更频率极低 |
| `category_id` | product | 支持按分类筛选预测结果 |
| `category_name` | product_category | 列表展示必需，变更频率极低 |

**查询对比**：

```sql
-- 无冗余时：需要JOIN 3张表
SELECT fr.*, p.name, p.product_code, pc.name as category_name
FROM forecast_result fr
LEFT JOIN product p ON fr.product_id = p.id
LEFT JOIN product_category pc ON p.category_id = pc.id
WHERE fr.forecast_date = '2026-03-23';

-- 有冗余后：单表查询
SELECT * FROM forecast_result
WHERE forecast_date = '2026-03-23';
```

### 3.2 进货建议表 `purchase_suggestion`

用于存储系统生成的进货建议，支持用户调整和管理。

```sql
-- 进货建议表
CREATE TABLE `purchase_suggestion` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- 关联信息
    `forecast_result_id` BIGINT DEFAULT NULL COMMENT '关联的预测记录ID',
    `product_id` BIGINT NOT NULL COMMENT '商品ID',

    -- ========== 冗余字段（减少JOIN查询） ==========
    `product_code` VARCHAR(50) NOT NULL COMMENT '商品编码(冗余)',
    `product_name` VARCHAR(100) NOT NULL COMMENT '商品名称(冗余)',
    `category_name` VARCHAR(50) DEFAULT NULL COMMENT '分类名称(冗余)',
    `purchase_price` DECIMAL(10,2) DEFAULT NULL COMMENT '进货单价(冗余，用于计算金额)',
    `confidence` INT DEFAULT NULL COMMENT '预测置信度(冗余，来自forecast_result)',

    -- 预测基础数据（快照）
    `predicted_quantity` INT NOT NULL COMMENT '预测销量',
    `current_stock` INT NOT NULL COMMENT '当前库存（生成建议时的快照）',
    `safety_stock` INT NOT NULL COMMENT '安全库存',
    `suggested_quantity` INT NOT NULL COMMENT '系统建议进货量',

    -- 用户调整
    `adjusted_quantity` INT DEFAULT NULL COMMENT '用户调整后的进货量，NULL表示未调整',
    `final_quantity` INT NOT NULL COMMENT '最终进货量（取adjusted_quantity或suggested_quantity）',

    -- 状态管理
    `status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态: 0待处理 1已生成进货单 2已忽略',
    `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注',

    -- 审计字段
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

    PRIMARY KEY (`id`),
    KEY `idx_forecast_result_id` (`forecast_result_id`),
    KEY `idx_product_id` (`product_id`),
    KEY `idx_status` (`status`) COMMENT '按状态筛选'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='进货建议表';
```

**冗余字段说明**：

| 冗余字段 | 来源表 | 理由 |
|----------|--------|------|
| `product_code` | product | 列表展示必需 |
| `product_name` | product | 列表展示必需 |
| `category_name` | product_category | 列表展示必需 |
| `purchase_price` | product | 计算进货金额必需（快照） |
| `confidence` | forecast_result | 列表展示预测置信度 |

**查询对比**：

```sql
-- 无冗余时：需要JOIN 4张表
SELECT ps.*, p.name, p.product_code, p.purchase_price, pc.name as category_name, fr.confidence
FROM purchase_suggestion ps
LEFT JOIN product p ON ps.product_id = p.id
LEFT JOIN product_category pc ON p.category_id = pc.id
LEFT JOIN forecast_result fr ON ps.forecast_result_id = fr.id
WHERE ps.status = 0;

-- 有冗余后：单表查询
SELECT * FROM purchase_suggestion WHERE status = 0;
```

### 3.3 进货单主表 `purchase_order`

用于记录正式的进货单，作为采购凭证。

```sql
-- 进货单主表
CREATE TABLE `purchase_order` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `order_no` VARCHAR(32) NOT NULL COMMENT '进货单号',

    -- 供应商信息
    `supplier_name` VARCHAR(100) DEFAULT NULL COMMENT '供应商名称',
    `supplier_contact` VARCHAR(50) DEFAULT NULL COMMENT '供应商联系方式',

    -- 订单信息
    `total_quantity` INT NOT NULL DEFAULT 0 COMMENT '商品总数量',
    `total_amount` DECIMAL(12,2) NOT NULL DEFAULT 0.00 COMMENT '进货总金额',

    -- 时间信息
    `order_date` DATE NOT NULL COMMENT '下单日期',
    `expected_date` DATE DEFAULT NULL COMMENT '预计到货日期',
    `actual_arrival_date` DATE DEFAULT NULL COMMENT '实际到货日期',

    -- 状态管理
    `status` TINYINT NOT NULL DEFAULT 0 COMMENT '状态: 0待确认 1已下单 2已完成 3已取消',

    -- 其他
    `remark` VARCHAR(500) DEFAULT NULL COMMENT '备注',
    `operator` VARCHAR(50) DEFAULT NULL COMMENT '操作人',

    -- 审计字段
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_order_no` (`order_no`),
    KEY `idx_order_date` (`order_date`),
    KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='进货单主表';
```

**状态简化**（相比之前的设计）：

```
待确认(0) → 已下单(1) → 已完成(2)
    ↓
已取消(3)
```

去掉"部分到货"状态，简化业务流程。

## 四、冗余字段汇总

### 4.1 各表冗余字段一览

| 表名 | 冗余字段 | 来源 |
|------|----------|------|
| `forecast_result` | product_code, product_name, category_id, category_name | product, product_category |
| `purchase_suggestion` | product_code, product_name, category_name, purchase_price, confidence | product, product_category, forecast_result |
| `purchase_order_item` | product_code, product_name, category_name | product, product_category |

### 4.2 冗余字段填充时机

```
┌─────────────────────────────────────────────────────────────────┐
│                        数据写入流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 执行预测时                                                   │
│     └─ 写入 forecast_result                                      │
│        └─ 同时查询 product + category，填充冗余字段              │
│                                                                 │
│  2. 生成进货建议时                                               │
│     └─ 写入 purchase_suggestion                                  │
│        └─ 从 forecast_result 获取 confidence                     │
│        └─ 从 product 获取 purchase_price                         │
│        └─ 从 forecast_result 冗余字段获取商品信息                 │
│                                                                 │
│  3. 生成进货单时                                                 │
│     └─ 写入 purchase_order_item                                  │
│        └─ 从 purchase_suggestion 冗余字段获取商品信息             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 冗余字段的一致性维护

**问题**：如果商品名称变更了，历史预测记录中的冗余字段怎么办？

**答案**：不需要维护！这正是冗余字段的意义——**保留历史快照**。

- 预测记录生成时的商品名称是"可口可乐500ml"
- 后来商品名称改成"可口可乐汽水500ml"
- 历史预测记录仍显示"可口可乐500ml"——这是正确的，记录的是当时的商品名称

## 七、业务场景SQL示例

### 场景1：执行预测并存储结果

```sql
-- 插入预测结果（含冗余字段）
INSERT INTO forecast_result (
    forecast_date, product_id,
    product_code, product_name, category_id, category_name,
    weather, is_holiday, is_weekend,
    predicted_quantity, confidence
)
SELECT
    '2026-03-23',
    p.id,
    p.product_code,
    p.name,
    p.category_id,
    pc.name,
    '晴', 0, 0,
    45, 85
FROM product p
LEFT JOIN product_category pc ON p.category_id = pc.id
WHERE p.id = 1;
```

### 场景2：查询预测结果列表（单表查询）

```sql
-- 有冗余后：单表查询，无需JOIN
SELECT
    id, forecast_date,
    product_code, product_name, category_name,
    predicted_quantity, confidence, actual_quantity, error_rate
FROM forecast_result
WHERE forecast_date = '2026-03-23'
ORDER BY category_name, product_name;
```

### 场景3：生成进货建议

```sql
-- 从预测结果生成进货建议（冗余字段一并复制）
INSERT INTO purchase_suggestion (
    forecast_result_id, product_id,
    product_code, product_name, category_name, purchase_price, confidence,
    predicted_quantity, current_stock, safety_stock,
    suggested_quantity, final_quantity
)
SELECT
    fr.id,
    fr.product_id,
    fr.product_code,
    fr.product_name,
    fr.category_name,
    p.purchase_price,
    fr.confidence,
    fr.predicted_quantity,
    p.stock,
    p.safety_stock,
    GREATEST(0, fr.predicted_quantity + p.safety_stock - p.stock),
    GREATEST(0, fr.predicted_quantity + p.safety_stock - p.stock)
FROM forecast_result fr
LEFT JOIN product p ON fr.product_id = p.id
WHERE fr.forecast_date = '2026-03-23'
  AND (fr.predicted_quantity + p.safety_stock - p.stock) > 0;
```

### 场景4：查询进货建议列表（单表查询）

```sql
-- 有冗余后：单表查询，无需JOIN
SELECT
    id, product_code, product_name, category_name,
    predicted_quantity, current_stock, safety_stock,
    suggested_quantity, adjusted_quantity, final_quantity,
    purchase_price, confidence, status
FROM purchase_suggestion
WHERE status = 0
ORDER BY suggested_quantity DESC;
```

### 场景5：预测效果评估

```sql
-- 预测日期过后，更新实际销量和误差率
UPDATE forecast_result fr
SET
    actual_quantity = (
        SELECT COALESCE(SUM(soi.quantity), 0)
        FROM sales_order_item soi
        WHERE soi.product_id = fr.product_id
        AND soi.sale_date = fr.forecast_date
    ),
    error_rate = CASE
        WHEN actual_quantity > 0
        THEN ROUND(ABS(predicted_quantity - actual_quantity) * 100.0 / actual_quantity, 2)
        ELSE NULL
    END
WHERE fr.forecast_date = '2026-03-23';
```
